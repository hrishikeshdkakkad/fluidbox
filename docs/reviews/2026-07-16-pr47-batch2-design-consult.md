## Numbered design verdicts

1. [lib.rs:1785](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-db/src/lib.rs:1785), [orchestrator.rs:157](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:157) — **AMEND.** Make `begin_finalization` one transaction that locks the session row, rejects terminal/missing sessions, computes `needs_quiesce` and deadline from that locked snapshot, then inserts-or-selects the winner. A plain INSERT followed by SELECT can race cleanup, and a delayed caller can recreate an intent after terminalization. Emit `QuiesceRequested` when this driver successfully applies active→Cancelling—not merely when `created`; the creator may crash before transition.

2. [fix-handover.md:29](/Users/hrishikeshkakkad/Documents/infra/docs/reviews/2026-07-16-pr47-fix-handover.md:29), [lib.rs:1847](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-db/src/lib.rs:1847), [workers.rs:39](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/workers.rs:39) — **AMEND.** Remove the status predicate entirely, not merely replace it with “NOT terminal.” A terminal session retaining an intent means post-terminal cleanup may still be owed. Recovery must scan every intent, treating terminal+intent as a cleanup/reconciliation work item. After any wind-down transition attempt, proceed only after confirming the expected state.

3. [orchestrator.rs:243](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:243), [internal.rs:793](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/internal.rs:793) — **AMEND.** `intent.needs_quiesce` is the semantic key; session status is still the phase guard. ## Defects the plan misses

- [orchestrator.rs:347](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:347), [orchestrator.rs:399](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:399), [orchestrator.rs:427](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:427) — Artifact, missing-artifact, and summary persistence errors are swallowed before terminalization. These writes must return errors and block the terminal transition; otherwise successful terminalization destroys the only recovery source.

- [lib.rs:1905](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-db/src/lib.rs:1905) — Re-drive is not semantically idempotent: a later transient collection failure can overwrite an already-collected good diff with `artifact_missing`. Preserve collected evidence over missing evidence, or skip collection once a durable collected artifact exists.

- [orchestrator.rs:355](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:355), [event.rs:201](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-core/src/event.rs:201) — Re-drive duplicates `run.result`, `artifact.collected`, and missing-artifact ledger entries because every retry receives a fresh event ID.

- [lib.rs:1726](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-db/src/lib.rs:1726), [orchestrator.rs:66](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:66), [lib.rs:2555](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-db/src/lib.rs:2555) — Terminal status commits before token revocation and delivery enqueue. A crash in that gap leaves a terminal session without those effects; delivery enqueue is not idempotent. Terminal-intent recovery must reconcile them under serialization before deleting the intent.

- [orchestrator.rs:34](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:34), [orchestrator.rs:40](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:40) — The new 30-second wait plus 120-second collection leaves little margin below the 180-second stale claim, while unbounded probes can exceed it. Overlapping drivers can overwrite evidence. Bound probes and either renew/release claims or increase the stale interval beyond the complete worst-case path.

- [orchestrator.rs:506](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:506), [orchestrator.rs:511](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/orchestrator.rs:511) — Active-status recovery can terminalize during provisioning before the handle lands; `set_sandbox_handle` can then attach a live sandbox to a terminal session. Batch 7’s periodic adoption/reconcile eventually addresses this, but batch 2 still needs a late-handle fence or must explicitly retain this known leak window.

- [internal.rs:162](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/internal.rs:162), [facade.rs:549](/Users/hrishikeshkakkad/Documents/infra/crates/fluidbox-server/src/facade.rs:549) — Returning `Result` from finalization is insufficient where detached budget tasks discard it. Tool-call budget finalization is particularly lossy because its one-shot verdict has already been committed; these callers need an explicit retry path.
