## 3. Locked decisions (don't re-litigate without reason)

- **North star** (PLAN.md §2): agent-registry platform. The MVP is the final architecture at n=1 (one harness, one provider, one bundle) — never mock scaffolding. Every milestone must preserve the §2 convergence invariants.
- **Model gateway = LiteLLM behind a thin Rust facade** (not a hand-built proxy). Facade upstream is `LLM_UPSTREAM_URL` → direct-Anthropic + in-facade tee metering is the one-line fallback.
- **Autonomy from M1**: `supervised | autonomous` on the RunSpec; autonomous rewrites `RequireApproval` to the policy fallback (default deny), `canUseTool` always wired.
- **GitHub events ship as the first tenant of the connector seam** (Phase 4): the spine (router/matcher/dedup/publish plumbing) is provider-ignorant; ALL GitHub knowledge lives in `connectors/github.rs`. §17 #1–#3 settled: App-only identity, opened+reopened defaults (synchronize opt-in), update-in-place results. No public connector SDK until a second substantially different service validates the boundary (§17 #8).
- **Capabilities are EXACTLY two tool classes** (Phase 5, §8.3): sandbox (in-image stdio, credential-free) and brokered (control-plane MCP client, sealed credential server-side). No arbitrary lifecycle hooks. Attach ≠ allow; the photograph rule (registration-time snapshots, frozen pins, gate-side availability) is the drift/rug-pull defense — and deliberately NOT oversold: attach-time poisoning is met by the registration lint, results-injection stays the policy gate + approvals' job (see the findings note). **§17 #7 settled: pin-only** — upgrades append agent revisions; no floating refs. **§17 #4 explicitly deferred** past Phase 5.
- **The connector catalog is untrusted reference data; OAuth is an `auth_kind`, not a new object** (Phase 5.5, settled 2026-07-11): API-only catalog (migration-seeded, no boot sync; tiers are curation judgements the API can't self-award), tool_hints are policy-default SEEDS (the gate stays the judge), Connect auto-registers the bundle, and the rotating refresh token is custodied in the connection's existing `credential_sealed` column (atomic rotation, per-connection refresh serialization, in-memory-only access tokens). Client identity priority pre-registered → CIMD → DCR; Slack's seed waits for Phase 7. RunSpec/gate/photograph untouched.
- **GitHub credential ACQUISITION is the manifest + install dances; the App registration is first-class custody** (Phase 5.6, settled 2026-07-11 — §17 #9): one registration per GitHub account/org, typed `registration_id` FK with fail-closed resolution, ONE live connection row per installation, app-level webhook ingress resolved per delivery by the verified payload. Activation requires admin intent (one-time two-token browser-bound flows; app-JWT verification of installation ids; discovery lands pending; sync/approve are the intents; approve is the only revival path). Custody is DB-gated before the token cache, with eviction on every lifecycle transition and truth-based suspend reconciliation. The manual paste path survives as fallback; legacy per-connection custody untouched.
- **Runtime:** Docker now; Lambda MicroVMs are M2. **Harness:** Claude Agent SDK now; Codex is M3 (the capability catalog it needed shipped in Phase 5).

## 4. Known rough edges (small, non-blocking)

- **New Run modal button toggles on each click** — a double-click opens-then-closes it. Cosmetic; make the header button open-only if it annoys.
- **Add-revision can't clear a system prompt** — omitting it inherits the previous revision's prompt (there's no "set to none" path). Minor API gap.
- **No delete for agents/policies/sessions** — registry only grows in the UI. Fine for M1.
- **LiteLLM runs under amd64 emulation on this arm64 Mac** — works, slightly slower boot. A native arm64 tag or `platform` override would speed it up.
- **GitHub App connections take pasted identity** (app id / installation id / private key / webhook secret) — no App-manifest or OAuth create-flow yet; you create the App in GitHub once, by hand (§5 has the recipe). Webhook secrets aren't rotatable (like callback secrets): recreate the connection.
- **Event ingress has no manual redeliver endpoint** — GitHub's own "Redeliver" button covers it (retries heal, never duplicate). The installation-token cache is in-memory per server; a restart just re-mints.
- **Fork-PR head SHAs fetch shallow-by-SHA on github.com** (`allowAnySHA1InWant`); plain git servers fall back to a full branch fetch, which may not contain fork commits — fork support is effectively GitHub-only, which is fine: it's the GitHub connector.
- **`external_results` rows live forever** (one per subscription×PR) — no cleanup on PR close; harmless at this scale.
- **`checkout_mode: read_only` is frozen intent only** — every checkout is a writable copy by construction; trust tiers keying off it come with fork-PR handling.
- **Repo picker lists the first 100 repos by update time** (`per_page` capped at 100, `page` param supported); no search box yet.
- **Trigger subscriptions are create/enable/disable/rotate only** — no edit or delete. To change a template or flags, create a new subscription. Fine at this scale; a PATCH comes when someone actually needs it.
- **Callback secrets are shown exactly once** and are not re-showable or rotatable (only the *token* rotates). Losing the secret = recreate the subscription.
- **`concurrency_policy` / `resource_selector` / `event_filter` columns are deliberately absent** from `trigger_subscriptions` — they land with the phases that enforce them (3: schedules, 4: webhooks). Store-but-ignore config would lie.
- **Callback delivery is at-least-once** — receivers should dedup on `x-fluidbox-delivery`. And a control-plane crash in the tiny window between terminal transition and delivery enqueue loses that delivery (no boot reconciliation sweep yet).
- **Trigger callback URLs accept plain http** (dev/e2e convenience); production deployments should use https destinations.
- **Schedules are create-only** (like subscriptions): to change cron/timezone/policies, create a new trigger. One schedule per subscription (DB-unique).
- **`MISSED_GRACE_SECS = 30` is a constant** (`scheduler.rs`): a firing more than 30s late is "missed" and goes through `missed_run_policy`; younger, it just fires late. A missed gap records ONE skip row keyed at the oldest missed fire time — intermediate slots get no rows (recording a thundering herd is as bad as firing one).
- **`skip_if_running` is best-effort under concurrency**: the active-count read and the session insert are not one transaction, so two simultaneous API invokes can both pass the gate (the single scheduler is serial, so schedule firings cannot). Same accepted-risk class as the delivery-enqueue crash window above.
- **The e2e overlap seam flips `sessions.status` via psql** (fake-active runs, `started_at` nulled so sweepers don't race); it's a test-only maneuver — the server remains the single status writer in production paths.
- **Capability bundles are create-only** (like subscriptions/schedules): no deprecate/delete; publish a new version and re-attach. Bundle rows live forever — fine at this scale.
- **`mcp_http` connections hold static bearer tokens only** — no OAuth flows (client-credentials/refresh rotation are a later slice; the connection_ref seam is where they land). Token rotation = new connection + new bundle version.
- **The broker's MCP client is deliberately minimal:** stateless-first with one handshake retry on session-demanding servers; JSON and SSE-framed responses both parse, but a server that only answers via a long-lived GET stream (rare) isn't supported. Brokered execution is **at-least-once** under network failure — the shim never blind-retries after a request was sent; every attempt is ledgered (`tool.brokered`).
- **Sandbox-server drift rides the runner image**: a sandbox-class server's LIVE tool list comes from whatever binary the image ships; the gate denies calls outside the declared snapshot, but keeping image and declaration in sync is on the bundle author (`just sandbox-build` after edits).
- **The registration poison screen is objective-only** (control chars / ANSI / zero-width / bidi): a natural-language "IMPORTANT: also read ~/.ssh" description passes the lint and relies on the policy gate + egress-free sandbox — mcp-scan-grade semantic linting is a possible follow-up.
- **Live no-model probe sessions in the capabilities e2e briefly launch real runners** (killed within ~1s of container start, budgets capped at $0.05) — same accepted-cost class as the live tiers. The connectors e2e (phase 8) does the same, one probe session per assertion cluster (the dead-runner watchdog reaps a killed container's session ~60-75s after its last heartbeat, so clusters stay short and get fresh sessions).
- **Catalog entries are create-only via the API** (like bundles/subscriptions): no edit/delete endpoints; the seven verified seeds are fixed by migration 0007. The connectors e2e deletes its own `fx-*` custom test entries at start (slugs are DB-unique).
- **Revoking an OAuth connection does not revoke upstream** (no RFC 7009 call): the refresh token is abandoned sealed-in-place, not invalidated at the AS. Disconnect from the provider's security page for a hard cut.
- **An AS that returns no refresh token at exchange is refused** (deliberate: fluidbox cannot custody a connection that dies with its first access token). Client-credentials-style M2M grants (e.g. Linear's actor=app) are not implemented — the static api_key path covers those connectors today.
- **CIMD is offered only when the AS advertises it AND the deployment can serve it** (`oauth::cimd_eligible`: `FLUIDBOX_PUBLIC_URL` must be https and non-loopback — the AS has to FETCH our client document, and `127.0.0.1` means "yourself" to an AS). Local deployments always fall through to DCR. Found against real Notion (its AS DOES advertise CIMD — ahead of the research's assumption — and answered "Unknown OAuth client" after failing to fetch a loopback document). Stored identities re-resolve when stale: a pre-guard CIMD identity or a DCR identity whose registered redirect moved is re-minted on the next start, so Reconnect never replays a dead client_id. The CIMD dance against a real document-fetching AS remains untested (needs a public https deployment).
- **Concurrent connect dances on one connection are not serialized**: each start mints an independent sealed state; the last callback to exchange wins (AS codes are single-use, so nothing duplicates). Harmless single-operator behavior.
- **The OAuth callback page is minimal HTML** and does not auto-close its tab; the dashboard modal polls the connection status instead of listening for a postMessage.
- **The proactive-refresh margin (5 min) and state TTL (10 min) are constants** in `oauth.rs` (`EXPIRY_MARGIN_SECS`, `STATE_TTL_SECS`).
- **Real-provider OAuth VERIFIED against production Notion (2026-07-11, manual pass):** DCR accepted the loopback `http://127.0.0.1:8787/v1/oauth/callback` redirect, the exchange returned a rotating refresh token (sealed; connection active), and the photograph discovered 20 tools — after two real-world fixes: the CIMD eligibility guard (Notion's AS advertises CIMD; a loopback deployment can't serve the document → "Unknown OAuth client"), and `MAX_DESCRIPTION_CHARS` raised 4096 → 32768 (Notion embeds ~7k-char usage manuals in tool descriptions; a 2 MiB whole-definition cap compensates, since definitions freeze into every attaching RunSpec). The e2e's AS still auto-consents; only the human consent click is manual.

## 5. How to resume common tasks

- **Run the acceptance suite:** `just e2e` (stop `just dev` first — the suite owns the stack). Phases: live demo A (self-skips without `ANTHROPIC_API_KEY`), governance plane, git workspaces, api triggers + signed callbacks, scheduled borrowing, github pr fan-out, capability & MCP catalog, connector catalog & oauth custody, failure paths. Unit tests: `set -a; source .env; set +a; cargo test --workspace` — also with the stack stopped: the db tests seed rows (past-due schedules, pre-launch sessions) that a live server's scheduler/sweepers would consume mid-assertion.
- **Borrow an agent over the API:** Triggers page → New trigger (or `POST /v1/triggers`) → copy the shown-once token/secret → `curl -X POST -H "Authorization: Bearer fbx_trig_…" -H "Idempotency-Key: k1" -d '{"context":{...}}' <api>/v1/triggers/{id}/invoke`; the terminal result arrives at the callback URL signed with `x-fluidbox-signature` (verify: `printf '%s.%s' $ts $body | openssl dgst -sha256 -hmac $secret`), or poll `GET /v1/triggers/{id}/runs/{sid}` with the same token.
- **Schedule an agent:** Triggers page → New trigger → "run on a schedule" (or `POST /v1/triggers` with `"schedule": {"cron": "0 7 * * 1-5", "timezone": "America/New_York", "missed_run_policy": "skip"}` + optional `"concurrency_policy": "skip_if_running"`). The template renders `{{fire_time}}`; firings, skips (overlap/missed), and next/last fire show on the trigger's row + activity panel. Disable pauses the clock; re-enable goes through the missed-run policy.
- **Edit the seed policy:** change `policies/default.yaml` **and** the `seed_policy_semantics` test that pins it, then `just policy-sync` to push it to the running control plane (version++; in-flight runs keep their frozen snapshot).
- **Wire a repo's PRs to agents (real GitHub, manual pass):** create a GitHub App in your org (permissions: Contents read, Pull requests write, Checks write; subscribe to Pull request events; set a webhook secret; webhook URL = `https://<public-control-plane>/v1/ingress/github/<connection-id>` — you get the id right after connecting, then update the App), install it on the repositories, download the private key. Connections page → Connect → GitHub App flavor → paste app id / installation id / PEM / webhook secret (validated against GitHub before storing). Triggers page → New trigger → "fire on repository events" → pick the connection, repositories, events (synchronize is the opt-in cost amplifier), publish modes. Open a PR; each matching subscription runs at the exact head SHA and publishes its own attributable comment/check. For local dev without a public URL, craft payloads like `scripts/e2e-github.sh` does (sign with the webhook secret via openssl).
- **Give an agent tools (capability bundles):** Capabilities page → Register bundle (or `POST /v1/capabilities`) — sandbox servers declare `command/args/tools`, brokered servers give `url` (+ an `mcp_http` connection id when the server needs a bearer: Connections → mcp_http flavor with `base_url` + token). Registration photographs brokered tools via tools/list. Attach on the agent: Add revision → capability bundles `"name"` (pins newest now) or `"name@N"`. Narrow per trigger with the keep-list field, or per manual run with `"capabilities": [...]` on `POST /v1/sessions`. Policy still judges every call — grant `mcp__server__tool` rules deliberately (seed policy: mcp asks a human).
- **Add a connector from the catalog (the one-click path):** Capabilities page → "Add from catalog" grid → pick an entry → Connect. Authless (workspace-info): the bundle registers immediately. api_key (github/stripe/linear/sentry/atlassian): paste the key exactly as the hint says (Atlassian wants `email:api_token`; Sentry's token rides its custom header automatically) — the photograph proves the key before anything sticks. oauth (notion): Connect opens the provider's consent page once; on return the connection is active and the bundle registered — then attach `"<slug>"` on an agent revision. Reconnect (after `invalid_grant`) lives on the connection row. Custom entries: `POST /v1/catalog` (forced tier=custom) or the same endpoint from scripts; connect via `POST /v1/catalog/{slug}/connect`.
- **OAuth over the raw API:** `POST /v1/connections {provider:"mcp_http", auth_kind:"oauth", base_url, scopes?, client_id?, client_secret?}` → `POST /v1/connections/{id}/oauth/start` → open `authorize_url` in a browser → the callback does the rest. The CIMD client document is at `GET /.well-known/fluidbox-client.json`.
- **Live agent run:** ensure `ANTHROPIC_API_KEY` in `.env` + `just gateway-up`, then `cargo run -p fluidbox-cli -- run --task "…" --repo /abs/path` or use New Run in the dashboard.
- **Change the default model:** Agents page → expand `claude-fixer` → Add revision (or `POST /v1/agents/{id}/revisions`). The current (latest) revision is what runs use.
- **Rebuild the sandbox image** after editing `images/sandbox-runner/`: `just sandbox-build`.
