- **Generic API borrowing + signed result callbacks (design-doc Phase 2, shipped 2026-07-10):** `trigger_subscriptions` (migration 0003) as the standing borrow instruction — **§17 #6 settled: caller task/workspace overrides are opt-in per subscription, both default OFF** (`allow_task_override`, `allow_workspace_override`); **scoped trigger tokens** (`api_tokens.kind='trigger'`, sha256-hashed, `fbx_trig_` prefix, rotation via `POST /v1/triggers/{id}/rotate_token`) that can invoke exactly one subscription + poll its runs and nothing else; **one unified `run_service::create_run`** every entry point (manual UI/CLI + API trigger) converges on, freezing `InvocationContext` (kind=manual|api, also written to `sessions.trigger`) and `result_destinations` into the RunSpec with wire-compat defaults for old rows; `POST /v1/triggers/{id}/invoke` with **Idempotency-Key** (claim-bind-release on `trigger_invocations`: replay returns the same run, body drift → 422, concurrent claim → 409, stale unbound claims stealable after 60s); strict `{{key}}` task templating from caller context; workspace narrowing (repo swap only within a github/connection base, ref/commit pins — never a new connection/URL/path); **signed webhook callbacks** — `result_deliveries` enqueued inside the single `orchestrator::transition` funnel on terminal entry, worker signs `x-fluidbox-signature: v1=hmac-sha256(secret, "{ts}.{body}")` with per-subscription AEAD-sealed secrets (shown once at creation), 6-attempt backoff 5s→1h, `callback.delivered/failed` ledger events, and a completed run stays completed when the destination is dead; scoped polling `GET /v1/triggers/{id}/runs/{sid}`; dashboard Triggers page (create + show-once token/secret + curl example, enable/disable, rotate, per-subscription runs & deliveries) plus invocation chip + deliveries panel on the session page.
- **Scheduled borrowing (design-doc Phase 3, shipped 2026-07-10):** a schedule is a **trigger subscription with a clock, never a new object** — `schedules` table (migration 0004: cron, **explicit IANA timezone** for DST-correct next-fire, `next_fire_at`, `missed_run_policy`, `last_fired_at`; one per subscription, enabled-ness rides the subscription) + a 1s tick worker (`scheduler.rs`, shaped like `deliveries.rs`) firing through the same `run_service::create_run` with `InvocationContext.kind=schedule` (cron/timezone/fire_time/catch_up frozen into the RunSpec). **Exactly-once firing**: each fire time claims a deterministic key (`sched:{fire_time}`) on the Phase 2 `trigger_invocations` table, and `create_session` binds the claim **in the same transaction** as the session insert — a crashed/restarted or double-fired scheduler replays, never duplicates. **§17 #5 settled (2026-07-10): overlap default `allow`, missed-run default `skip`; `concurrency_policy` lives on `trigger_subscriptions` and is enforced in `create_run` for ALL invocations** — `skip_if_running` records a visible skip (claim row `skip_reason='overlap'`; API invokes get 409), `replace` cancels the active run (reason names the replacement) then starts fresh. Missed gaps (scheduler down or subscription disabled — same path) record ONE skip keyed at the oldest missed fire time, or fire exactly ONE `catch_up` run — never fire-all-missed. Config-time validation (cron/tz parse, template must render from `{{fire_time}}` alone, policies enumerated); pure cron domain in `fluidbox-core/src/schedule.rs` (`cron` + `chrono-tz`, 5-field normalized, 6-field seconds = the e2e fire-fast seam). Dashboard: schedule line on trigger rows (cron/tz/next/last fire, missed policy, concurrency chip), firings-&-skips column, schedule create form.
- **GitHub PR-review fan-out on the connector seam (design-doc Phase 4, shipped 2026-07-10):** the CONNECTOR SEAM is the feature; GitHub is its first tenant. One **provider-ignorant event spine** (`events.rs`: ingress → verify → normalize → match → `create_run` → publish — the e2e greps it to prove no provider name appears) with ALL GitHub knowledge in **one connector module** (`connectors/github.rs` behind the plain-match dispatch in `connectors/mod.rs`; no SDK/trait registry — §17 #8 n=1 discipline; Slack later = one new module + one `ResultDestination` variant). **Two-level DB-unique idempotency** (migration 0005): `trigger_deliveries` unique(connection, external_event_id) + `trigger_dispatches` unique(delivery, subscription), the dispatch bound to its session in the same transaction — webhook retries *heal* partial fan-outs, never duplicate runs or comments. **GitHub App connections** (`provider=github_app`: app id + installation id + RS256 private key + webhook secret, all validated then AEAD-sealed; installation tokens minted+cached ~1h; PAT connections stay for fetch); ingress `POST /v1/ingress/github/{connection_id}` is deliberately unauthenticated — the `X-Hub-Signature-256` HMAC against the sealed secret IS the auth, and nothing is stored before it verifies. **Fan-out**: one PR event → one run per matching subscription (event filter + repository selector, fail-closed matcher), each frozen at the **exact head SHA** with `InvocationContext.kind=event` (provider/external_event_id/event_type/resource frozen for audit); §17 #5 concurrency policies govern event dispatch too. **Fork trust tier is real**: head repo ≠ base repo (or hidden — fails toward fork) freezes `TrustTier::ReadOnly` + `checkout_mode=read_only`, enforced at the permission gate by `policy::read_only_denial` (allowlist: read tools + metachar-free read-only shell) ABOVE policy/subscription/human approval — no approval escape; denials ledger as `source=trust_tier`. **Publishers** ride the Phase-2 `result_deliveries` worker as new `ResultDestination` variants: `github_pr_comment` (one stable comment per (subscription, PR) via `external_results`, PATCHed in place on later events — §17 #3; recreated only if deleted externally) and `github_check` (one run per head SHA named `fluidbox/<subscription>`), both under the **App identity** (§17 #1) with per-agent attribution in the content; one agent's failure shows only on its own comment/check. **§17 #1–#3 settled 2026-07-10** (recorded in the design doc): App-only; default events `opened`+`reopened` with `synchronize` per-subscription opt-in (cost amplifier); update-in-place. Dashboard: App connect form + ingress URL on connection rows, event-subscription form (events, repos, publish modes) + event badges. E2E needs no public URL: fake GitHub API via `FLUIDBOX_GITHUB_API_URL` + `file://` clone base via `FLUIDBOX_GITHUB_CLONE_BASE`.

## 4. Known rough edges (small, non-blocking)

- **New Run modal button toggles on each click** — a double-click opens-then-closes it. Cosmetic; make the header button open-only if it annoys.
- **Add-revision can't clear a system prompt** — omitting it inherits the previous revision's prompt (there's no "set to none" path). Minor API gap.
- **No delete for agents/policies/sessions** — registry only grows in the UI. Fine for M1.
- **LiteLLM runs under amd64 emulation on this arm64 Mac** — works, slightly slower boot. A native arm64 tag or `platform` override would speed it up.
- **GitHub App connections take pasted identity** (app id / installation id / private key / webhook secret) — no App-manifest or OAuth create-flow yet; you create the App in GitHub once, by hand (§5 has the recipe). Webhook secrets aren't rotatable (like callback secrets): recreate the connection.
- **Event ingress has no manual redeliver endpoint** — GitHub's own "Redeliver" button covers it (retries heal, never duplicate). The installation-token cache is in-memory per server; a restart just re-mints.
- **Fork-PR head SHAs fetch shallow-by-SHA on github.com** (`allowAnySHA1InWant`); plain git servers fall back to a full branch fetch, which may not contain fork commits — fork support is effectively GitHub-only, which is fine: it's the GitHub connector.
- **`external_results` rows live forever** (one per subscription×PR) — no cleanup on PR close; harmless at this scale.
- **`checkout_mode: read_only` is frozen intent only** — every checkout is a writable copy by construction; trust tiers keying off it come with fork-PR handling.
- **Repo picker lists the first 100 repos by update time** (`per_page` capped at 100, `page` param supported); no search box yet.
- **Trigger subscriptions are create/enable/disable/rotate only** — no edit or delete. To change a template or flags, create a new subscription. Fine at this scale; a PATCH comes when someone actually needs it.
- **Callback secrets are shown exactly once** and are not re-showable or rotatable (only the *token* rotates). Losing the secret = recreate the subscription.
- **`concurrency_policy` / `resource_selector` / `event_filter` columns are deliberately absent** from `trigger_subscriptions` — they land with the phases that enforce them (3: schedules, 4: webhooks). Store-but-ignore config would lie.
- **Callback delivery is at-least-once** — receivers should dedup on `x-fluidbox-delivery`. And a control-plane crash in the tiny window between terminal transition and delivery enqueue loses that delivery (no boot reconciliation sweep yet).
- **Trigger callback URLs accept plain http** (dev/e2e convenience); production deployments should use https destinations.
- **Schedules are create-only** (like subscriptions): to change cron/timezone/policies, create a new trigger. One schedule per subscription (DB-unique).
- **`MISSED_GRACE_SECS = 30` is a constant** (`scheduler.rs`): a firing more than 30s late is "missed" and goes through `missed_run_policy`; younger, it just fires late. A missed gap records ONE skip row keyed at the oldest missed fire time — intermediate slots get no rows (recording a thundering herd is as bad as firing one).
- **`skip_if_running` is best-effort under concurrency**: the active-count read and the session insert are not one transaction, so two simultaneous API invokes can both pass the gate (the single scheduler is serial, so schedule firings cannot). Same accepted-risk class as the delivery-enqueue crash window above.
- **The e2e overlap seam flips `sessions.status` via psql** (fake-active runs, `started_at` nulled so sweepers don't race); it's a test-only maneuver — the server remains the single status writer in production paths.

## 5. How to resume common tasks

- **Run the acceptance suite:** `just e2e` (stop `just dev` first — the suite owns the stack). Phases: live demo A (self-skips without `ANTHROPIC_API_KEY`), governance plane, git workspaces, api triggers + signed callbacks, scheduled borrowing, github pr fan-out, failure paths. Unit tests: `set -a; source .env; set +a; cargo test --workspace` — also with the stack stopped: the db tests seed rows (past-due schedules, pre-launch sessions) that a live server's scheduler/sweepers would consume mid-assertion.
- **Borrow an agent over the API:** Triggers page → New trigger (or `POST /v1/triggers`) → copy the shown-once token/secret → `curl -X POST -H "Authorization: Bearer fbx_trig_…" -H "Idempotency-Key: k1" -d '{"context":{...}}' <api>/v1/triggers/{id}/invoke`; the terminal result arrives at the callback URL signed with `x-fluidbox-signature` (verify: `printf '%s.%s' $ts $body | openssl dgst -sha256 -hmac $secret`), or poll `GET /v1/triggers/{id}/runs/{sid}` with the same token.
- **Schedule an agent:** Triggers page → New trigger → "run on a schedule" (or `POST /v1/triggers` with `"schedule": {"cron": "0 7 * * 1-5", "timezone": "America/New_York", "missed_run_policy": "skip"}` + optional `"concurrency_policy": "skip_if_running"`). The template renders `{{fire_time}}`; firings, skips (overlap/missed), and next/last fire show on the trigger's row + activity panel. Disable pauses the clock; re-enable goes through the missed-run policy.
- **Edit the seed policy:** change `policies/default.yaml` **and** the `seed_policy_semantics` test that pins it, then `just policy-sync` to push it to the running control plane (version++; in-flight runs keep their frozen snapshot).
- **Wire a repo's PRs to agents (real GitHub, manual pass):** create a GitHub App in your org (permissions: Contents read, Pull requests write, Checks write; subscribe to Pull request events; set a webhook secret; webhook URL = `https://<public-control-plane>/v1/ingress/github/<connection-id>` — you get the id right after connecting, then update the App), install it on the repositories, download the private key. Connections page → Connect → GitHub App flavor → paste app id / installation id / PEM / webhook secret (validated against GitHub before storing). Triggers page → New trigger → "fire on repository events" → pick the connection, repositories, events (synchronize is the opt-in cost amplifier), publish modes. Open a PR; each matching subscription runs at the exact head SHA and publishes its own attributable comment/check. For local dev without a public URL, craft payloads like `scripts/e2e-github.sh` does (sign with the webhook secret via openssl).
- **Live agent run:** ensure `ANTHROPIC_API_KEY` in `.env` + `just gateway-up`, then `cargo run -p fluidbox-cli -- run --task "…" --repo /abs/path` or use New Run in the dashboard.
- **Change the default model:** Agents page → expand `claude-fixer` → Add revision (or `POST /v1/agents/{id}/revisions`). The current (latest) revision is what runs use.
- **Rebuild the sandbox image** after editing `images/sandbox-runner/`: `just sandbox-build`.

## 6. Logical next steps

Ordered by leverage. The seams for all of these already exist (that's the point of the M1 architecture) — none require reworking the core.

### A. Near-term hardening — ✅ DONE 2026-07-10
All three items shipped (plan: `docs/superpowers/plans/2026-07-10-6a-hardening.md`):
1. **`just e2e`** — one command, three phases, 45 checks: `scripts/e2e.sh` orchestrates `e2e-live.sh` (demo A: live agent fixes a failing test → completed + diff + cost + isolation), `governance-e2e.sh`, `e2e-failures.sh`, over a shared `e2e-lib.sh`.
2. **Failure paths automated** (`scripts/e2e-failures.sh`): tool-call budget stop, dead-container watchdog, restart orphan sweep (reaps unknown-session containers, spares live sessions), **plus a newly-found gap fixed**: sessions stuck in `created`/`provisioning`/`initializing` after a control-plane crash are now failed by a 15-min stale-launch sweep (`Created→Failed` edge added; two real zombie rows in Neon got cleaned by it).
3. **PLAN §10 #1 + #3 resolved** from ledger data: classifier tuned (read-only utils allowed; any force-push spelling and split-flag `rm -rf /` denied) and budgets set to 1800 s / 1 M tokens / $2.50 / 100 tool calls — pinned by `seed_policy_semantics`. Bonus fix: `policy.budgets` was parsed but never read; it's now a **ceiling** at session creation. Live DB policy synced (v4).

### B. M2 — Lambda MicroVM provider + BYOC (the flagship runtime) — the big one
Implement `LambdaMicrovmProvider` behind the existing `ExecutionProvider` trait. This is where "customer-owned execution" becomes real. Key work (all validated in PLAN.md §2/§7):
- `RunMicrovm` + image pipeline (S3 zip → snapshot), in-VM supervisor serving the runner contract, JWE token minting/refresh inside `EndpointHandle` (60-min TTL → refresh loop), 8-hour lease rollover + `Harness::checkpoint()` in the orchestrator, idle-suspend/resume for long-running autonomous agents.
- Terraform BYOC module + S3 workspace/artifact store.
- **Why it's the seam it is:** `SandboxHandle` is already serializable jsonb and the `initializing` workspace phase is provider-agnostic, so nothing above the trait changes. Prove it by running the *same* demo A on the MicroVM provider with only a `runs_on: lambda-microvm` toggle.

### C. M3 — Multi-harness + capability catalog (completes the north star)
- **Codex runner:** second first-party image (`codex mcp-server` via rmcp, or the TS Codex SDK as payload). OpenAI key custody moves into the same LiteLLM gateway. This proves the harness abstraction with n=2.
- **Capability-bundle catalog:** tool/MCP bundles as first-class registry objects with credential brokering; UI to attach/narrow them per revision. Every MCP call policy-gated like any tool.
- **Customer-signed runner images** — the "bring your own agent" endgame.
