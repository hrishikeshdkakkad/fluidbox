# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**fluidbox** is a control plane that runs AI coding agents in governed, disposable sandboxes. A user registers a versioned **agent definition**; each **run** freezes an immutable `RunSpec`, provisions a fresh sandbox, streams a live event timeline, pauses for human approval (or auto-decides in autonomous mode), and ends with a diff + cost report. `PLAN.md` is the authoritative design doc and roadmap — read it before making architectural changes. It defines convergence invariants (§2) that every change must preserve.

Hard constraints (from PLAN.md, non-negotiable): fluidbox-authored backend is 100% Rust; the Next.js dashboard is presentation-only (all logic in the Rust API); database is Neon Postgres; OSS/MIT.

## Commands

```bash
just setup          # fresh-clone bootstrap (idempotent): .env + secrets, apps/web/.env.local, pnpm install, runner image
just doctor         # preflight — validates every env gotcha below and prints the exact fix
just dev            # LiteLLM gateway + server + dashboard together
just server         # Rust control plane only (migrations auto-run on boot)
just web            # dashboard only (Next.js, port 3000)
just gateway-up     # start the pinned LiteLLM container (reads .env)
just sandbox-build  # rebuild the sandbox runner image after editing images/sandbox-runner/
just check          # fmt + clippy -D warnings + test + web build (the full bar)
just e2e            # full acceptance: live demo A + governance + git workspaces + api triggers + schedules + github fan-out + capability catalog + connector catalog/oauth + failure paths (owns the stack; stop `just dev` first)
just policy-sync    # push policies/*.yaml to the running control plane (version++)

cargo test -p fluidbox-core                              # fast, no DB needed
cargo test -p fluidbox-db                                # needs DATABASE_URL (real Neon)
cargo test -p fluidbox-core policy::tests::               # a single module's tests
cargo run -p fluidbox-cli -- run --task "…" --repo /path # drive a run from the CLI
```

The `fluidbox-db` and workspace-wide test runs hit **real Neon** — `set -a; source .env; set +a` first so `DATABASE_URL` is present, otherwise the DB test self-skips.

## Environment & setup gotchas (these cost real debugging time)

- **`FLUIDBOX_BIND` must be `0.0.0.0:8787`, not loopback.** Sandboxes reach the control plane via `host.docker.internal`, which resolves to the host's gateway IP — a `127.0.0.1` bind is unreachable from a container.
- **`FLUIDBOX_DATA_DIR` is canonicalized to an absolute path at startup** (`config.rs`). Docker bind mounts reject relative paths; don't undo this.
- **Neon: use the DIRECT (non-`-pooler`) connection string.** PgBouncer transaction mode breaks sqlx prepared statements and `LISTEN/NOTIFY` (`PgListener` needs a direct connection). `scripts/neon-setup.sh` enforces this.
- **The Anthropic key lives ONLY in the LiteLLM container**, injected via docker-compose from `.env`. The Rust server never holds it (it authenticates to LiteLLM with `LITELLM_MASTER_KEY`). No server restart is needed after adding the key — just `just gateway-up`.
- **`.env` is gitignored; `apps/web/.env.local` too** (it carries the admin token for the dashboard proxy). Never commit either.
- sqlx needs the `macros` + `derive` features; clap needs `env`; reqwest 0.13 uses the `rustls` feature (not `rustls-tls`). LiteLLM is pinned by **digest** in `.env` (`LITELLM_IMAGE=...@sha256:...`); tag `main-v1.91.1` does not exist as an image — use `main-stable` and re-pin the digest.
- **`FLUIDBOX_CREDENTIAL_KEY` (32-byte hex/base64) gates integration connections AND event ingress.** Without it the server boots fine but Connections are disabled. It seals GitHub PATs, App private keys, and webhook secrets at rest (`seal.rs`, the v1 legacy format). *Losing* it orphans stored credentials, but **retirement now has a supported path** (Phase D): enable KMS envelope sealing (`FLUIDBOX_KMS_MODE`), run the re-seal job (`POST /v1/admin/reseal`), then retire the legacy key once boot proves zero v1 rows (losing `FLUIDBOX_CREDENTIAL_KEY` before that re-seal orphans the remaining v1 rows). Custody then roots on the KEK: **losing the KEK is unrecoverable** from the moment any v2 row exists, and after retirement it loses ALL custody — back it up (see the sealing invariant + `docs/hosted/kms-operations.md`). Git fetch credentials pass to git via `GIT_CONFIG_*` env vars only (never argv, never on-disk config — the `.git` dir is mounted into the sandbox).
- **`FLUIDBOX_PUBLIC_URL` is the browser/AS-facing base URL** (default `http://127.0.0.1:8787`): it feeds the OAuth `redirect_uri` (`/v1/oauth/callback`) and the CIMD client-id document (`/.well-known/fluidbox-client.json`). Neither can ride `FLUIDBOX_PUBLIC_CONTROL_URL` — that one resolves to host.docker.internal for sandboxes, which browsers and authorization servers can't reach.
- **`FLUIDBOX_GITHUB_API_URL` + `FLUIDBOX_GITHUB_CLONE_BASE` are the GitHub test seams.** The e2e points the first at a local fake API and the second at a `file://` fixture root, so the whole PR fan-out path (incl. App-token minting and comment/check publishing) runs with no public URL and no real GitHub.
- **Multi-user identity is off by default; `FLUIDBOX_REQUIRE_SSO=1` turns it on** (Phase B). It confines the admin token to `/v1/admin/*` (break-glass only); browsers then authenticate via `__Host-fbx_web` session cookies (`fbx_web_` token) and CLIs via `fbx_pat_` PATs — both sha256-only at rest, and all three of `fbx_sess_`/`fbx_web_`/`fbx_pat_` are scrubbed by the `event.rs` Redactor (there's a test). Session/OIDC knobs: `FLUIDBOX_SESSION_IDLE_SECS` (28800), `FLUIDBOX_SESSION_ABSOLUTE_SECS` (604800), `FLUIDBOX_SESSION_REAUTH_SECS` (60, clamped ≤60), `FLUIDBOX_OIDC_DISCOVERY_MAX_AGE_SECS` (3600), `FLUIDBOX_OIDC_CLOCK_SKEW_SECS` (60); `FLUIDBOX_TRUST_FORWARDED_FOR` (default off — client IP is the socket peer otherwise).
- **The dashboard proxy's credential mode is `FLUIDBOX_WEB_MODE=admin|sso`** (static per deployment, default `admin`): `admin` injects the admin token server-side (today's local/single-admin behavior); `sso` carries NO admin token — it forwards the `__Host-fbx_web` cookie + `x-fluidbox-csrf` header only. Cookie-authenticated non-GETs require the CSRF header + a same-origin `Origin`; there is no CORS layer (same-origin proxy).
- **`FLUIDBOX_KMS_MODE` (`off` default | `static` | `aws`) turns on envelope sealing** (Phase D). `static` needs `FLUIDBOX_KMS_STATIC_KEK` (32-byte, local/CI only); `aws` needs `FLUIDBOX_KMS_AWS_KEY_ID` (+ optional `FLUIDBOX_KMS_AWS_ENDPOINT`, a test seam) — boot fails naming a missing one. The legacy-key retirement gate is bidirectional (KMS-off refuses if any v2 row exists; legacy-key-absent refuses if any v1 row remains); don't drop `FLUIDBOX_CREDENTIAL_KEY` until the re-seal job proves zero v1. Runbook: `docs/hosted/kms-operations.md`.
- **`FLUIDBOX_LLM_KEY_MODE` (`shared` default | `tenant`) picks how the facade authenticates upstream** (Phase D). `tenant` mints a per-tenant LiteLLM virtual key and confines the master key to provisioning — but it REQUIRES LiteLLM backed by its OWN Postgres (`/key/*` is a DB-backed LiteLLM feature); none of the shipped composes/charts wire this, so local stays `shared`. Hosted (`FLUIDBOX_REQUIRE_SSO=1`) + `shared` ⇒ the facade `503 tenant_llm_keys_required` on every call (boot warns). Tuning: `FLUIDBOX_LLM_ADMIN_URL` (defaults to `LLM_UPSTREAM_URL`), `FLUIDBOX_LLM_TENANT_MODELS|MAX_BUDGET|BUDGET_DURATION|TPM|RPM`.
- **The egress knobs are `FLUIDBOX_EGRESS_ALLOW_CIDRS` + `FLUIDBOX_EGRESS_PROXY`** (Phase E, both optional, both **fail boot** on a malformed value). The first opts specific CIDRs back in past the shared SSRF predicate (a deliberately-private LiteLLM/GHES/MCP endpoint); the second routes BOTH hardened reqwest clients *and* the git fetch subprocess through one waypoint — and moves target DNS resolution to the proxy, so the resolver's name-filtering stops applying to proxied requests (`admit_url`'s literal+scheme checks still do): point it at an allowlisting forward proxy. Outbound abuse control: `FLUIDBOX_EGRESS_RATE_{TENANT,CONNECTION,HOST}_PER_MIN` (120/60/120) and `FLUIDBOX_EGRESS_BREAKER_{THRESHOLD,OPEN_SECS}` (5/60) — `0` DISABLES that dimension (never "block everything"), all of it is **per-replica in-memory** (deployment ceiling = N× the value; durable limiting is Phase F). `FLUIDBOX_LLM_MAX_CONCURRENT_RESERVATIONS` (32) caps concurrent per-run LLM budget reservations.
- **`FLUIDBOX_RUNTIME_ROLE` (unset default; the Helm chart now defaults it to `fluidbox_runtime`) opts into the RLS non-owner pool split** (Phase D). Set it to `fluidbox_runtime` (migration 0018's NOLOGIN role) and every pooled connection `SET ROLE`s to it — boot fails if the role doesn't exist, and re-runs 0018's POSTURE validation on every boot: refusals for `carries unsafe attribute(s)` (LOGIN/SUPERUSER/BYPASSRLS/CREATEROLE/CREATEDB/REPLICATION), `is a member of` (privileges we never granted), `is granted to` (another principal can `SET ROLE` in). The **migration** reads the name too (session GUC `fluidbox.runtime_role`), so it's ONE ROLE NAME PER DEPLOYMENT on a shared cluster — roles are cluster-global while the grants are database-local. Unset = single-role mode (RLS still binds even the owner via `FORCE`) — but a SUPERUSER/BYPASSRLS role (Neon's default) skips RLS entirely and attributes are NOT inherited through membership, so under `FLUIDBOX_REQUIRE_SSO=1` that combination is a **boot refusal** (`REFUSING TO BOOT: FLUIDBOX_REQUIRE_SSO=1 (multi-user) but the database role this pool runs as …`); single-user only warns, `FLUIDBOX_ALLOW_RLS_BYPASS=1` opts out (local single-user only — it makes 0018 decoration), and `just doctor` fails on the unbootable combination. `SET ROLE` is NOT a credential boundary (`RESET ROLE` restores the owner; the process still holds the owner `DATABASE_URL`) — it contains our own bugs, nothing more. 0018 is **stop the old binary, migrate, then deploy** (a pre-0018 binary sets no GUC ⇒ zero rows, AND holds transactions across outbound HTTP that would block the migration's ACCESS EXCLUSIVE locks).

## Architecture — how a run flows

The crate dependency order is `fluidbox-core` → `fluidbox-db` / `fluidbox-provider` → `fluidbox-server`; `fluidbox-cli` is a thin reqwest client.

- **`fluidbox-core`** — pure domain, no I/O. The policy engine (`policy.rs`), session state machine (`state.rs`), canonical event schema + redaction (`event.rs`), `RunSpec`/autonomy (`spec.rs`), and the extension traits (`traits.rs`: `ExecutionProvider`, `Harness`). Change domain rules here and the tests here.
- **`fluidbox-server`** is one binary with several planes wired in `main.rs`:
  - **Public `/v1` API** (`api.rs`) — the `auth.rs` principal resolver: Operator (admin token) | User (browser `__Host-fbx_web` cookie) | Pat (`fbx_pat_` bearer), RBAC in `rbac.rs`; the dashboard + CLI talk only here.
  - **Identity `/v1/auth/*`** (`login.rs`, `tokens.rs`) — per-org IdP-agnostic OIDC login (two-phase, browser-bound one-time `login_flows`), logout, `/v1/auth/me`, PAT mint/list/revoke.
  - **Org admin `/v1/admin/orgs*`** (`admin_orgs.rs`) — org + IdP-config lifecycle, break-glass owner arming, membership roles, issuer migration; the only surface the admin token reaches under `FLUIDBOX_REQUIRE_SSO`.
  - **Internal gateway `/internal`** (`internal.rs`) — per-session-token auth; **only the in-sandbox runner talks here**. Houses the permission handler (the heart of the system) and `events`/`heartbeat`/`result`.
  - **LLM facade `/internal/llm`** (`facade.rs`) — the sandbox's fake `ANTHROPIC_API_KEY` **is its session token**; the facade validates it, enforces the budget stop, swaps in the real upstream credential, forwards to LiteLLM, and tees the SSE stream to meter usage.
  - **Orchestrator** (`orchestrator.rs`) — drives lifecycle transitions; the **server is the single status writer**, the runner only reports.
  - **Workers** (`workers.rs`) — heartbeat watchdog, wall-clock budget sweeper, approval expiry, boot-time orphan reap.
  - **SSE** (`sse.rs`) — the event stream.
- **`images/sandbox-runner/runner/index.mjs`** is the **only sanctioned Node payload**: the Claude Agent SDK harness. It implements the runner contract (`canUseTool`→`/permission`, message stream→`/events`, heartbeats, final `/result`). It is sandbox *workload*, not backend Rust — that's why Node here doesn't violate the Rust constraint.

### Load-bearing invariants (violating these breaks the security/audit model)

- **RunSpec is frozen at session creation** and stored as jsonb, including a full policy snapshot. In-flight runs are governed by their snapshot; editing an agent or policy only affects *future* runs. This immutability is what makes the audit trail trustworthy.
- **Tenant isolation is a signature requirement, not a remember-to-filter convention** (Phase B). Every tenant-owned `fluidbox-db` method takes a `TenantScope` (constructible only from a verified tenant) and carries its id into a `tenant_id = $n` predicate; the principal-less loaders (global scans, `get_session`/`get_connection`/… by bare id) live ONLY in `fluidbox-db::system_worker`, and there are exactly two credential-like bootstrap exceptions — token-digest resolution (you don't know the tenant until the token resolves) and the pending-switch confirmation.
- **Tenant isolation also has a DATABASE floor** (Phase D, migration 0018): 37 tenant-owned tables `ENABLE`+`FORCE` RLS; policies key on `current_setting('fluidbox.tenant_id')` (child tables via parent `EXISTS`; four specials bespoke). `FORCE` binds even the table owner (the reason Phase B deferred it). `fluidbox-db::scoped_tx` sets the tenant GUC transaction-locally; the audited cross-tenant bypass is `worker_tx` (sets `fluidbox.bypass='system_worker'` — the category rides IN the GUC value). `worker_tx` is `pub(crate)`, so the server crate cannot construct a bypass ad-hoc; it reaches one only through a NAMED, `pub` `fluidbox-db` entry point — MOSTLY but NOT ONLY in `system_worker` (`reseal_begin`, `global_registration_tx`, … hand out a bypass-armed tx; outside that module `identity::insert_audit_standalone` is `pub` and takes the bypass path when `entry.tenant_id` is `None`). The in-code inventory in the `system_worker` module docs is the authority — keep it complete; the property is "a short, named, grep-able set of escape hatches", not "unreachable" and not "only `system_worker`". Two tables split read from write: `connector_catalog` + `oauth_client_registrations` hold deployment-GLOBAL rows (`tenant_id NULL`) that any scope may `SELECT` but only the bypass may `INSERT/UPDATE/DELETE`. `auth_audit_log`'s INSERT policy carries the SAME floor (tenant GUC match or bypass) — a GUC-less audit insert is refused, so an audit outside a mutation's tx goes through `identity::insert_audit_standalone`; `audit_select` stays tenant-or-null-or-bypass. `FLUIDBOX_RUNTIME_ROLE` splits the pool so connections `SET ROLE` to the NON-owner `fluidbox_runtime` (grants are ENUMERATED per table — never `ALL IN SCHEMA` — minus `auth_audit_log` UPDATE/DELETE, the 0012 deferral); the identity/bindings/secrets e2e suites set it, so the whole HTTP surface runs RLS-enforcing there. **RULE: a NEW tenant-owned table needs all three in its own migration — `ENABLE`+`FORCE` RLS, a policy, and an explicit DML grant to the deployment's runtime role (resolve it from `current_setting('fluidbox.runtime_role')`, never hardcode `fluidbox_runtime`)** — there is deliberately no `ALTER DEFAULT PRIVILEGES`, and 0018's drift guard ABORTS the migration if a table it policies is missing from the enumerated grant list. A SUPERUSER or BYPASSRLS role (Neon's `neon_superuser`, CI's `postgres`) skips RLS entirely and role attributes are NOT inherited through membership, so the negative tests connect as `fluidbox_runtime`, multi-user boot REFUSES a bypassing pool role (`FLUIDBOX_ALLOW_RLS_BYPASS=1` opts out), and `just doctor` inspects the role the SERVER will actually run as — the configured `FLUIDBOX_RUNTIME_ROLE` (name/existence/posture, then `current_user` behind a `SET ROLE`), else the `DATABASE_URL` role — failing on every state boot refuses and warning otherwise. **Checking the `DATABASE_URL` role in runtime-role mode is the wrong question** and misreports both ways (a BYPASSRLS Neon owner + a correct runtime role is ENFORCING; an RLS-bound owner + an absent/unsafe runtime role will not boot). **0018 is stop-the-old-binary, migrate, then deploy** — an old binary sets no GUC (zero rows) and holds transactions across outbound HTTP.
- **Custody sealing is VERSIONED envelope encryption** (Phase D, #32; `seal.rs`/`kms.rs`): every sealed column carries a `<col>_key_version` companion — `1` = legacy `nonce‖ct` under `FLUIDBOX_CREDENTIAL_KEY`; `2` = `[0x02][dek_version][nonce][ct]` under a per-tenant DEK wrapped by a KEK (`FLUIDBOX_KMS_MODE=off|static|aws`). The AAD binds `fbx:v2:{tenant_id}:{table.column}`, so a blob is untransplantable across tenants OR columns. There are THIRTEEN sealed families (`SealFamily`, incl. the `connector_oauth_flows` PKCE verifier); ephemeral transit tokens (OAuth/login/GitHub-App `state`) seal under the boot tenant's DEK. Boot runs TWO retirement gates that fail closed BOTH directions. Legacy→v2 is the resumable, CAS-guarded, count-parity `POST /v1/admin/reseal` singleton (audit rows); the legacy key retires only when boot proves zero v1. Runbook: `docs/hosted/kms-operations.md`.
- **Agents are append-only.** Editing an agent = appending a `agent_revision` (never mutating one). The model + system prompt live on the revision; a run uses the *current* (latest) revision's values.
- **Two distinct prompts:** the **system prompt** is on the agent revision (who the agent is); the **task** is per-run (what to do this time). The New Run flow only takes the task.
- **The permission callback stays wired in both autonomy modes** — never the SDK's `bypassPermissions`. Autonomous mode rewrites a `RequireApproval` verdict to the policy fallback *inside* `evaluate()`, recording both the original and rewritten verdict in the ledger.
- **Approvals are idempotent by `(session_id, tool_call_id)`.** The DB row is the source of truth; the in-memory `Notify` only wakes a blocked handler early. On restart, the runner's retry re-attaches to the pending row — nothing duplicates or hangs.
- **The ledger only accepts `Redacted<EventEnvelope>`** (`event.rs`) — constructible solely via `Redactor::scrub`. Model prompts never reach the ledger; only digests + usage + cost. `append_event()` (SQL function) assigns a gapless per-session `seq` under a row lock and `pg_notify`s.
- **SSE fanout is hybrid:** NOTIFY is only a wakeup; the `seq` catch-up query is the delivery source of truth (immune to missed notifies and Neon scale-to-zero). Same query powers `Last-Event-ID` resume.
- **Workspace init is control-plane-side.** The credentialed fetch/copy happens in the orchestrator during the `initializing` state (before the agent starts); the agent only ever sees a bind-mounted copy at `/workspace`. The original repo is never touched, and the sandbox stays egress-free.
- **Trigger tokens are subscription-scoped** (`api_tokens.kind='trigger'`, sha256-hashed): a token can invoke its one subscription (`POST /v1/triggers/{id}/invoke`) and poll the runs it created — never the admin API, and the admin token can never invoke. Invoke task/workspace overrides are **opt-in per subscription** (design doc §17 #6, default off) and can only narrow authority (repo/ref/commit inside the subscription's github/connection base; never a new connection, clone URL, or local path). All entry points converge on `run_service::create_run`, which freezes the `InvocationContext` + result destinations into the RunSpec.
- **Result delivery is decoupled from the run lifecycle** (`deliveries.rs`): rows are enqueued inside the single `orchestrator::transition` funnel on terminal entry (exactly-once by the state machine); the retry worker signs payloads `v1=hmac-sha256(secret, "{ts}.{body}")` with per-subscription AEAD-sealed secrets and backs off 5s→1h over 6 attempts. A callback failure can never mutate a session; delivery is at-least-once — receivers dedup on `x-fluidbox-delivery`.
- **A schedule is a trigger subscription with a clock, never a new object** (`schedules`, migration 0004; `scheduler.rs`). The tick worker fires through the same `run_service::create_run` with `InvocationContext.kind=schedule`; firing is **exactly-once** via a deterministic idempotency claim (`sched:{fire_time}` on `trigger_invocations`) that `create_session` binds in the same transaction as the session insert. §17 #5 (settled 2026-07-10): `concurrency_policy` defaults `allow` and is enforced in `create_run` for ALL invocations (`skip_if_running`/`replace` are opt-in; an API invoke against an active run gets a 409 skip); missed-run defaults `skip` — a gap records ONE skip row keyed at the oldest missed fire time, `catch_up` fires exactly ONE make-up run, never fire-all-missed. Skips are terminal claim rows (`skip_reason`), visible per subscription. A disabled subscription's schedule does not advance — re-enabling goes through the missed-run path.
- **Capabilities are EXACTLY two tool classes, and the split IS the security model** (design §8.3, Phase 5; **brokered plumbing rebuilt in Phase C #31**): *sandbox* MCP servers are stdio subprocesses packaged in the runner image (credential-free, contained by the container); *brokered* MCP servers are called BY THE CONTROL PLANE (`broker.rs`) with a sealed credential that never enters a sandbox — same inversion as the LLM facade and git fetch. **`capability_bundles` now survives for SANDBOX stdio tools ONLY** — `capabilities.rs::register_bundle` REFUSES any `class:brokered` server (cutover error → docs/guides/capabilities.md). Brokered tools moved to four objects: connector definition (catalog) → **connection** (owns the credential + append-only `connection_tool_snapshots`; `snapshots.rs` forced-`initialize` photograph, `protocol_version` recorded, cursor-cap fail-closed; created at connect + OAuth activation + `POST /v1/connections/{id}/tools/refresh`; read via `GET /v1/connections/{id}/tools`) → agent-revision **`connection_requirements`** (`{slot, connector{url,slug}, required_tools, binding_mode invoking_user|organization}`, `capability.rs::validate_requirements`, satisfaction:ALL fail-closed) → per-run **`run_resource_bindings`** (`bindings.rs::resolve_run_bindings` resolves each requirement to a tagged authority `connection|subscription_secret|none`, slot_kinds `mcp|workspace_fetch|result_publish`, BEFORE provisioning, written in `create_session`'s tx; `RunSpec.brokered` surfaces + `binding_id` on workspace/destinations; explicit per-slot `bindings` on `POST /v1/sessions`, unknown slots rejected). **§17 #7 (settled 2026-07-10): sandbox bundles still PIN exact versions at attach** ("name" = latest THEN, "name@N" explicit); nothing floats. `run_service::create_run` freezes brokered surfaces (binding_id + snapshot_version + tools + digest) + sandbox schema snapshots + digests, after remove-only keep-list intersection (§3.5) and stripping ALL mcp on `TrustTier::ReadOnly`. **Attach ≠ allow**: the ONE gate (`internal.rs::decide_tool_call`, `/permission` + `/internal/sessions/{id}/tools/call`) checks budget → **frozen-set availability** (drifted/rug-pulled = deny `source=capability`; missing/stale binding = deny `source=binding`) → **frozen-schema args** (Phase E, `source=schema`) → trust tier → policy → approvals — Phase C only grew the frozen set into a binding-backed union; Phase E inserted exactly ONE new stage (schema) and moved nothing. Before ANY brokered secret access, `broker.rs::recheck_binding` (+ orchestrator workspace-fetch + deliveries publish consumers) reverify status + `authorization_generation` + (personal connections) owner-membership-active — fail-closed. **`authorization_generation` bumps on reconnect of ever-activated OAuth connections** (rotation within a generation is fine; GitHub App lifecycle NEVER bumps — installation identity proven); stale-generation bindings refuse mid-run. **Approval authority: a personal-connection call is decidable ONLY by its owner-who-invoked — NO role/admin/operator override** (`rbac.rs::authorize_approval_decision`, symmetric approve/deny). Ledger per brokered call: `tool.requested` → `tool.decision` → `tool.brokered` (latency + result digest, never payloads/secrets); the runner auto-allows brokered `mcp__*` in canUseTool because the broker runs the identical gate server-side. **Legacy residual (honest):** pre-Phase-C in-flight RunSpecs keep the embedded-`connection_id` broker path with STATUS-ONLY recheck (no binding/generation to compare); historical RunSpecs untouched; a revision still pinning a brokered bundle is REFUSED at create_run ("Phase C" cutoff), and 0013 appends converted revisions + repoints pinned subscriptions (mixed brokered+sandbox bundles drop WHOLE with a raise-notice). §17 #4 (first brokered git-writes) still deferred; rides this gateway.
- **The connector catalog is UNTRUSTED reference data; OAuth custody rides the existing connection object** (Phase 5.5, settled 2026-07-11): `connector_catalog` is seeded by migration 0007 and managed API-only (`/v1/catalog` — no seed file, no boot sync; custom entries FORCED tier=custom); `tool_hints` are policy-default seeds for display — the gate stays the judge. **Phase C #31: custom rows are now TENANT-SCOPED** (curated/imported stay global `tenant_id null`; `list_catalog`/`get_catalog_by_slug` take scope, a tenant row shadows a same-slug global; 0013 backfills the single boot tenant else DISABLES via `disabled_at`; partial unique indexes = global slug + per-tenant custom slug; adding a custom entry needs admin/owner, personal-custody Connect open to any member). Catalog Connect for a REMOTE server now photographs a `connection_tool_snapshots` row, NOT a bundle (api_key = seal + `snapshots::photograph_connection`, rollback on refusal; oauth = the dance, callback photographs); only the `none`/in-image path still `register_bundle`s a sandbox bundle. Connections also gained `owner_type`/`owner_user_id`/`created_by_user_id` (personal = owner-only visible/mutable, admins excluded by design; membership deactivation is the kill switch). **The refresh token lives in `credential_sealed`** discriminated by `auth_kind` (`static`|`oauth`|`none` — `none` = credentialless remote, Phase C) — the active-only reader covers OAuth custody for free, and rotation is ONE atomic UPDATE (old token dead the moment the new one lands; refreshes serialize per connection via `oauth_locks`). Access tokens are minted at call time and cached in memory ONLY (`connector_tokens`) — restart re-mints; proactive refresh <5 min to expiry + ONE reactive-401 retry (the 401 proves the tool never executed; static credentials that 401 are terminal). The dance: 401 → RFC 9728 PRM → RFC 8414/OIDC (refuse without PKCE S256) → authorize with S256 + RFC 8707 `resource=` on BOTH legs → `GET /v1/oauth/callback` (**unauthenticated by design** — the AEAD-sealed `state` param carrying connection_id+verifier IS the auth, like webhook signatures) → exchange requires a refresh token → seal → active. Client identity priority: pre-registered (sealed `client_secret_sealed`, confidential clients) → CIMD (`/.well-known/fluidbox-client.json`, also public — its URL IS the client_id; **only when `FLUIDBOX_PUBLIC_URL` is https + non-loopback** — the AS must be able to fetch the document, so local deployments always use DCR) → DCR (minted id stored per connection, never re-registered while valid; stale identities — ineligible CIMD, moved redirect — re-resolve on the next start). `invalid_grant` ⇒ `status='error'` ⇒ create_run/photograph/broker all fail closed off the status; reconnect re-runs the dance on the SAME connection. `FLUIDBOX_PUBLIC_URL` (browser/AS-facing base) feeds redirect_uri + CIMD. RunSpec/gate/photograph are UNTOUCHED — only `broker::brokered_auth` resolution grew (custom `header_name`/`scheme` on static connections — the Sentry/Atlassian shapes — and OAuth minting).
- **Connector OAuth is now a ONE-TIME, browser-bound flow with REUSABLE client identities** (Phase D, invariant 20; `oauth.rs`, migrations 0015/0016). `POST /v1/connections/{id}/oauth/start` (and catalog Connect) returns ONLY a `go_url` (`/v1/oauth/go?f=…`) — the browser navigates it, which sets the `__Host-fbx_oauth_flow` cookie and claims a one-time `connector_oauth_flows` row whose `browser_hash` sits INSIDE the atomic single-use predicate (a leaked authorization URL can neither complete nor burn a flow). The row FREEZES authorization/token endpoints + resolved client + `resource` + sealed PKCE verifier + `expected_generation` at start; the callback exchanges against the ROW (closing AS mix-up) and refuses a moved generation. The old `seal_state`/`open_state` stateless helpers are GONE. Client identity is a shared `oauth_client_registrations` row per `(issuer, redirect_uri)` (global rows in v1), DCR singleflighted by a Postgres advisory lock, adopt-verbatim on reuse, with exchange-only `invalid_client` self-heal (re-register + retry); per-connection pre-registered custody survives. The flow cookie is **always `Secure`** — the `__Host-` prefix is *defined* as Secure + `Path=/` + no Domain, and every conforming client (browsers, and curl ≥7.87) DISCARDS a `__Host-` cookie that lacks it, so the old "omit Secure on local http" special-case silently deleted the cookie and the callback always answered "this browser did not start the connect flow". **Residual:** a NON-loopback plain-http `FLUIDBOX_PUBLIC_URL` still can't complete Connect (browsers refuse `Secure` off a non-trustworthy origin; loopback http is fine, and a real AS requires https anyway); and the **`go_url` is TRANSFERABLE** — the `/go` boot token carries the flow-cookie plaintext (`c`) and `/go` is what SETS the cookie, so whichever browser opens the link becomes "the initiating browser". The binding therefore defeats a leaked AUTHORIZATION url, not a deliberately shared START url: an attacker starts Connect on their own pending connection, lures a victim into consenting, and the victim's refresh grant seals into the ATTACKER's connection. **A connected-account confirmation closes this ONLY if it gives the EXTERNAL ACCOUNT HOLDER informed consent — shown to the browser COMPLETING the flow, or binding an expected external subject captured at start; a confirmation shown to the flow's INITIATOR closes nothing, because in the lure the initiator IS the attacker and would simply confirm the victim's account.** (Earlier revisions of this file and of the threat model claimed the deferred confirmation UI closed it — corrected 2026-07-20, #32.) The known closure is to move the browser-facing leg onto the origin that already holds the session: configurable callback URI → dashboard origin behind the existing proxy in hosted mode, flow cookie set on the AUTHENTICATED start response, proxy forwards it to the control plane, and `c` DROPPED from the `/go` token (`__Host-` cookies are host-locked and in `sso` mode the proxy hands `Set-Cookie` back to the browser, so the session cookie lives on the DASHBOARD origin while `/go` + `/v1/oauth/callback` live on the CONTROL-PLANE one). Not built — needs a proxy cookie-allowlist change and a registered dashboard callback URI; full write-up in `docs/hosted/threat-model.md`.
- **The LLM facade authenticates PER-TENANT when asked** (Phase D; `facade.rs`, `llm_keys.rs`, migration 0017): `FLUIDBOX_LLM_KEY_MODE=shared` (default: one deployment key) | `tenant` (a per-tenant LiteLLM virtual key, minted lazily, sealed like any §custody column under the tenant's own DEK, cached UNSEALED in memory, never in a response or a sandbox). In `tenant` mode the master key is confined to `llm_keys.rs` provisioning — a routine model request never rides it (in the default `shared` mode the facade DOES present the deployment key, i.e. `LITELLM_MASTER_KEY` against a LiteLLM upstream, on every model request). Fail-closed refusals: `FLUIDBOX_REQUIRE_SSO=1` + `shared` ⇒ `503 tenant_llm_keys_required`; a tenant with no mintable key ⇒ `503 tenant_llm_key_unavailable`; boot refuses `shared`+empty-key or `tenant` without a LiteLLM master key/upstream. It is the fairness backstop, not the per-run budget-race fix — that is Phase E's reservations (below). Rotate: `POST /v1/admin/orgs/{slug}/llm-key/rotate`.
- **ALL control-plane egress rides ONE hardened boundary, and admission happens TWICE** (Phase E, Gap 7; `egress.rs`, `netpolicy.rs` in core). Two hardened clients: `state.egress_http` (broker MCP + discovery + delivery callbacks) refuses redirects OUTRIGHT — `Policy::none`, and a bare 3xx *status* is refused identically, `Location` digested at debug, never followed — and `state.identity_http` (per-hop re-validation) carries OIDC plus connector-OAuth **discovery** (PRM, AS metadata, DCR) — but **both connector-OAuth TOKEN legs deliberately ride the no-redirect `egress_http`** (code exchange + stored-bag refresh), because a 307/308 REPLAYS the request BODY, forwarding an authorization code + PKCE verifier or a refresh token to the redirect target; the flow froze `token_endpoint` at start, so a redirect there means the endpoint moved mid-flow and is refused outright rather than same-origin-restricted (source-grep test pins it). GitHub/facade/`llm_keys` deliberately stay on the plain `state.http` as operator-configured seams. Both use the filtering DNS resolver, but **reqwest dials an IP LITERAL without ever consulting a resolver**, so the pure `egress::admit_url` pre-flight (scheme policy + host-literal class) is what actually stops `https://169.254.169.254/…` — it fronts EVERY dial site (`broker::dial_rpc`, deliveries publish, all six oauth fetches incl. the stored-bag refresh). Blocked classes live in `fluidbox-core::netpolicy` (private/loopback/link-local/multicast/reserved/benchmarking/documentation/v6-site-local/metadata — metadata blocked EVEN under the dev seam); `FLUIDBOX_EGRESS_ALLOW_CIDRS` opts back in, `FLUIDBOX_EGRESS_PROXY` re-points both clients + git. Admission ALSO runs at SAVE time (connection base_url on both `mcp_http` create paths, subscription `callback_url` ⇒ 400) — literal+scheme only, no DNS in a handler. Git is out-of-process so it gets its own `GitEgressPolicy`: scheme allowlist, `http` dev-seam-only, `file://` ONLY under `FLUIDBOX_GITHUB_CLONE_BASE` or the dev seam, resolve-and-validate EVERY address, plus `-c http.followRedirects=false` + `GIT_LFS_SKIP_SMUDGE=1` + `GIT_ALLOW_PROTOCOL=http:https:file` asserted against the production argv/env builders. **Disclosed residuals:** the clone path is resolve-THEN-validate and git re-resolves independently (TOCTOU); a configured proxy owns DNS, so name-filtering stops applying to proxied requests.
- **Upstream MCP is a PER-RUN session with a conformance contract, and version drift denies the call** (Phase E, Gap 8; `broker.rs`, `mcp_sse.rs`). A replica-local registry keyed `(run session, peer)` holds the upstream session id + negotiated version + request-id counter — never shared across runs or users (invariant 11). `initialize` + `notifications/initialized` precede every call (the stateless-first path is DELETED), `MCP-Protocol-Version` rides every post-initialize request, and the authorization header rides EVERY request **including the terminal `DELETE`**, whose credential is re-resolved live through `recheck_binding`→`brokered_auth_for_conn` (never cached on the registry entry — inv 9 + inv 22) and whose DELETE is SKIPPED when that resolution fails. We offer `2025-11-25` with `SUPPORTED={2025-11-25, 2025-06-18}`; at run time `check_negotiated` requires EXACT equality with the frozen surface's `protocol_version` when present (drift ⇒ deny, message names `POST /v1/connections/{id}/tools/refresh`), else supported-set membership. SSE is a real incremental assembler (per-event 256 KiB cap = error, 8 MiB decoded total); server→client REQUESTS get `-32601` posted back, notifications are ignored (`list_changed` logged, never auto-refreshed); a 404-with-session reinitializes ONCE in the same slot. `outputSchema`/`structuredContent` are PRESERVED (schema folded into the tool digest only when present so old digests are byte-identical; structured content relayed + digested) — **results are NOT validated against `outputSchema`** (disclosed). SEP-835 `insufficient_scope` on a 401/403 is TERMINAL (no re-mint), sets `status='error'` with a reconnect note + token eviction — and because it is a definitive upstream response it classifies `failed_upstream`, so a replay of the same `tool_call_id`+input adopts the stored refusal while a fresh model turn proceeds normally. **Residual:** the registry is replica-local with no sweeper — a run finalized on a different replica leaves entries there until process exit (session affinity is Phase F).
- **Frozen schemas are ENFORCED server-side, with the dialect chosen by the snapshot's protocol version** (Phase E, Gap 12, invariant 17; `fluidbox-core::schema_guard`, `internal.rs::schema_gate_decision`). The schema is UNTRUSTED input (invariant 13) so it is pre-guarded before compilation — ≤256 KiB serialized, depth ≤32, every `$ref` local (`#…`); a violation makes the TOOL un-callable rather than being ignored — and arguments are bounded (≤1 MiB, depth ≤64) with a deny-everything external-`$ref` retriever. `dialect_for`: `2025-11-25` ⇒ 2020-12 (SEP-1613), any other recorded version ⇒ draft-07, `None` (surfaces frozen before `BrokeredSurface.protocol_version`, and sandbox bundles) ⇒ 2020-12. Compiled validators sit in a hand-rolled LRU (cap 256) keyed `(tools_digest, tool)` with the dialect re-verified on hit. Placement is exact: after frozen-set availability, BEFORE trust tier, `mcp__*` only, denial reusing the byte-identical capability CAS-then-ledger path with `source="schema"` so the model sees a tool-execution error (SEP-1303 shape), never a protocol error.
- **Every brokered dispatch is wrapped in a durable four-state execution claim** (Phase E, Gap 11, invariants 15/16/18; migration 0019 `tool_execution_claims`). UNIQUE `(session_id, tool_call_id, input_digest)` — the digest is the gate's, threaded down rather than recomputed, so a reused id with different arguments is a NEW claim, never an adoption. The claim tx takes `select status from sessions … for update` FIRST and reads terminality from that LOCKED row (same lock order as cancellation and `set_sandbox_handle`; never held across HTTP), then `INSERT … ON CONFLICT DO NOTHING RETURNING`. States: `claimed → succeeded | failed_upstream | failed_before_send | ambiguous`. **`failed_before_send` requires POSITIVE proof nothing was written** (admission refusal, binding/auth failure, breaker-open, `reqwest::is_connect()`) and is the ONLY re-claimable state (same row, `attempt+1`); a definitive upstream response — HTTP error status, JSON-RPC error, `isError` — is `failed_upstream` and TERMINAL; timeout/mid-stream/post-connect redirect refusal is `ambiguous`, as is a `claimed` row the 10s sweeper finds past expiry. A loser adopts the stored outcome verbatim; ALL Definitive outcomes render `{ok:true, result:{content, is_error}}` to the runner while the LEDGER keeps `ok = !is_error` — protocol convention and audit truth are deliberately distinct.
- **The sandbox holds FOUR audience-scoped tokens, not one bearer** (Phase E, Gap 10, invariant 19; migration 0020). `llm|tool|control|workspace` are separate `api_tokens` rows (kind stays `session`, all `fbx_sess_`-prefixed so the Redactor is unchanged) with `audience text NOT NULL DEFAULT 'all'` — that DEFAULT is PERMANENT, not an artifact: it keeps in-flight sessions and the e2e psql forgers working. `auth.rs::require_audience` is the FIRST statement of every guarded handler and answers `403 {"error":"wrong_audience"}` — the BODY CODE is load-bearing (the runner keys its fatal abort on it, so changing the body silently restores silent-deny). Mapping: permission/tools-call ⇒ `tool`; events/heartbeat/result/token-renew ⇒ `control`; workspace ⇒ `workspace`; facade ⇒ `llm` (checked BEFORE session load, budget, and dispatch). `/result` checks the audience BEFORE the revoked-token leniency, so a revoked control token still ACKs while an llm/tool token never does. K8s ships ONE Secret with FOUR keys routed per container (init sees only workspace); both runners `delete process.env.FLUIDBOX_SESSION_TOKEN` before ANY spawn. **Residuals:** same-uid children can still read the runner's INITIAL env via `/proc/<pid>/environ` (the scrub covers the spawned env only — the invariant-19 acceptance bullet is met, true process isolation is a follow-up); Docker `HostDev` is explicitly not a boundary; and an OLD pinned image on a NEW server is UNSUPPORTED — current runner-lib aborts loudly (`EXIT_AUDIENCE_MISMATCH`, diagnostic on the timeline, no `/result`), but that behavior lives IN the image, so a pre-Phase-E image still collapses to silent denial while model spend continues. The route→audience mapping is proven ONLY by the e2e matrix — a wrong constant is invisible to every Rust unit test (proven by deliberate mutation).
- **Approval emission is single because it rides the decision CAS; lifecycle and delivery are fenced** (Phase E, Gap 13; migration 0021). `approval.decided` and `tool.decision` are appended INSIDE the deciding transaction (`decide_approval_tx` / `expire_approval_tx` / the terminal-deny claim), so only the CAS winner emits and waiters emit NOTHING; a `pg_notify` on the new `fluidbox_approvals` channel in the same tx wakes waiters on every replica, with the ≤2s poll floor kept as the missed-notify backstop. Consequence worth keeping in mind: a ledger-append failure now rolls the DECISION back (fail-closed, deliberate). Lock order is BINDING — approvals→sessions at the decision sites, sessions→claims/reservations everywhere else; the pairs are disjoint so no cycle exists. Sessions carry `orchestrator_{owner_id,lease_until,epoch}`: the epoch bumps ONLY on owner CHANGE (re-taking our own lapsed lease deliberately does NOT self-fence — self-fencing would kill any launch slower than the TTL), driver mutations run through `transition_session_fenced`, while request-side intents (cancel, forced finalize) stay UNFENCED so a user's cancel never depends on who holds the lease. Deliveries claim rows `FOR UPDATE SKIP LOCKED` and re-stamp `claimed_until` per row immediately before each attempt (`DELIVERY_CLAIM_TTL_SECS` = 300 is DERIVED from the worst-case single publish, not magic); the GitHub double-post window closes by reconcile-before-create on BOTH external surfaces — comments on a deterministic `<!-- fluidbox:sub:… -->` body marker (real defense: the unguessable UUIDv7 plus a trailing-token position check — NOT that the searched body is free of agent text), checks on the app-controlled `external_id` field listed per head SHA (#33; the display name is user text and any app can publish one, so it is never the adoption key). A reconciliation ERROR is NOT proof-of-absence: it FAILS the attempt so the retry re-reconciles — a delayed comment beats a duplicate, which cannot be repaired. Checks, like comments, refuse a subscription-less publish (no subscription, no stable identity, no idempotence). Delivery stays at-least-once (receivers dedup on `x-fluidbox-delivery`) and a crashed claimant's rows park for up to the TTL. A fenced-out driver still runs `finish_terminal_cleanup` on purpose — cleanup is the retry ticket and is idempotent; do NOT tighten that.
- **LLM budget admission is a durable, request-keyed reservation** (Phase E, Gap 14; migration 0022 `llm_reservations`). The row's pk IS the request UUID that later becomes `usage_entries.external_id`, which is what makes the 401 replay and a late drain idempotent. Admission is `select … from sessions … for update` FIRST and then ONE CTE (the CTE alone does NOT close the race under READ COMMITTED — two txs each snapshot before the other commits), booking a deliberately-HIGH estimate — declared max output + `body_len / BYTES_PER_INPUT_TOKEN` where that constant is **1**, i.e. one byte per token, a genuine upper bound rather than an average (priced when the model is known). Release happens ONLY on positively-proven non-dispatch — the release set R1-R13 is enumerated in a comment above the booking and a future early return MUST join it; a stream that drains with zero usage markers RETAINS (never assume zero). Charging requires a DURABLE usage write first: `record_usage` must succeed before the CAS to `charged`, else the row stays `reserved` and the sweeper converts it into a conservative `source='reservation_timeout'` charge. `RESERVATION_TTL_SECS` (1800) must comfortably EXCEED `UPSTREAM_HTTP_TIMEOUT_SECS` (900) — a test derives the assertion from the constant rather than hardcoding it. **Two disclosed deviations:** the SOLE-CLAIMANT carve-out (budget arms skipped when zero reservations are active, so one request can proceed over budget — bounded to budget + one request's actual usage, i.e. no worse than pre-Gap-14, and it prevents a 429 livelock where a lone oversized request could never start), and the sweeper's aggressive projection (counting live reservations can stop a run that would have fit: ~$0.82 of a $2.50 default budget for an opus-4-class run with a request in flight, ~6.6% for haiku). The terminal "out of budget" verdict still comes from the accumulated check + the sweeper.
- **GitHub credentials are ACQUIRED via the manifest + install dances; the App registration is a first-class custody object** (Phase 5.6, settled 2026-07-11; design `docs/plans/2026-07-11-github-seamless-connect-design.md`). `github_app_registrations` custodies the app identity (sealed pem/webhook/client secrets; one registration per GitHub account/org — private apps install only on their owner); seamless connections carry a typed `registration_id` FK with **fail-closed** custody resolution (registration missing/not-active = refuse, never fall back to per-connection custody), and a partial unique index guarantees ONE live row per installation (revoked rows revive only via `POST /v1/connections/{id}/approve`, keeping the connection id — and dedup history — continuous). **Activation requires admin intent**: admin-token'd start endpoints mint one-time `github_app_flows` rows driven by TWO sealed tokens (bootstrap `gh-boot` for the go page; `gh-manifest`/`gh-install` states for the GitHub round-trip — never interchangeable), a per-flow HttpOnly cookie whose hash sits INSIDE the one-time claim predicate (a leaked URL can neither complete nor burn a flow), and `installation_id` is NEVER trusted — it must resolve under OUR app's JWT (`GET /app/installations/{id}`). GitHub-initiated discovery (`installation.created` webhooks, state-less setup hits) can only create `pending` rows — state-less setup does ZERO writes and ZERO GitHub calls; sync/approve are the admin intents. **App-level ingress** (`/v1/ingress/github/app/{registration_id}`, HMAC vs the registration's sealed secret) resolves the connection from the verified payload's installation id and feeds the SAME `events::process_delivery` spine (raw-body digests; ping/unknown installations → 202 acks; events.rs stays grep-clean). Custody is **DB-gated, not cache-gated**: `installation_token` fresh-reads connection + registration status BEFORE the token cache may serve; suspend/unsuspend reconcile against GitHub truth (webhook order never wins), delete → revoked, registration revoke cascades to children — all with token eviction. Lifecycle DB failures 5xx (GitHub retries) — never swallow-and-ack. `FLUIDBOX_GITHUB_WEB_URL` is the browser-facing GitHub seam (manifest form target, install URLs); trace spans log method+path only (query strings carry codes/states). The manual paste path (`POST /v1/connections`) survives as the fallback; legacy per-connection custody rows are untouched.
- **Connected-service events ride ONE provider-ignorant spine** (`events.rs`: ingress → verify → normalize → match → `create_run` → publish); ALL GitHub knowledge lives in `connectors/github.rs` behind the `connectors/mod.rs` dispatch — the only provider→module mapping, and deliberately a plain match, not a trait registry/SDK (§17 #8). The e2e greps `events.rs`/`run_service.rs` for provider names; keep them clean. Ingress (`POST /v1/ingress/{provider}/{connection_id}`) is deliberately unauthenticated — the webhook signature against the connection's sealed secret IS the auth, and nothing is stored before it verifies. **Two DB-unique dedup levels** (`trigger_deliveries` unique(connection, external_event_id); `trigger_dispatches` unique(delivery, subscription), bound to the session in the same transaction) make webhook retries HEAL partial fan-outs — never duplicate runs or comments. **Fork PRs freeze `TrustTier::ReadOnly`** (head repo ≠ base repo, or hidden — fails toward fork): enforced at the permission gate via `policy::read_only_denial` ABOVE policy/subscription/human approval (no approval escape). §17 #1–#3 (settled 2026-07-10): results publish **App-only** (attribution in content; checks need App identity anyway); default events **opened+reopened** with `synchronize` an explicit opt-in (it fires per push); one stable comment per (subscription, PR) **updated in place** via `external_results`, one check per head SHA named `fluidbox/<subscription>`. Publishing rides `result_deliveries` — a dead GitHub can never mutate a run.

## Extension points

New execution backend → implement `ExecutionProvider` (the Lambda MicroVM provider is the next planned one; `SandboxHandle` is already serializable jsonb for reattach). New agent harness → **a new runner image implementing the HTTP runner contract** (`/permission`, `/events`, `/heartbeat`, `/result` + the broker shim's `/tools/call`; the env contract; the CANONICAL tool vocabulary — names/shapes crossing `/permission` MUST be `Bash{command}`, `Edit/Write/MultiEdit{file_path | edits[].file_path}`, `Read/Glob/Grep/LS`, `mcp__<server>__<tool>` — canonicalization is runner-side) **plus one arm in the `harness.rs` registry** (id validation, image/model defaults, per-harness env extras — a plain match, §17 #8, NOT a trait registry; there is deliberately no `Harness` trait). There are TWO runner images today: `images/sandbox-runner/` (Claude Agent SDK) and `images/codex-runner/` (Codex app-server, the second harness — Phase 6), sharing `images/runner-lib/` (the runner-contract client + broker/sandbox-gate shims). The seams exist precisely so nothing above them changes — see PLAN.md §6.2.
